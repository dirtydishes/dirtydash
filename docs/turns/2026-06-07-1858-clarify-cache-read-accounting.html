Clarify Cache Read Accounting

ScopeDashboard cache metrics
Issuedirtydash-e2f
Validationbuild, tests, smoke

Summary

Renamed the dashboard cache metric from a full-sounding hit ratio to an observed cache read share, added explicit copy for unreported cache writes, and exposed a clearer API field while preserving the old name for compatibility.

Changes Made

  • Added cache_read_share to the summary cache payload and kept hit_ratio as a compatibility alias.
  • Changed Overview and Cache page labels from hit-rate language to observed-read-share language.
  • Changed the cache-write metric to Reported writes with a zero-state note when logs do not expose writes.
  • Added an Accounting note on the Cache page explaining that Dirtydash only counts cache lifecycle tokens present in local logs.
  • Adjusted README wording to frame cache accounting as log-reported evidence.

Context

Local Codex session logs expose cached_input_tokens, which is enough to report cached input reads. They do not appear to expose positive cache creation/write tokens in the sampled local data. Calling the observed read share a cache hit ratio made the UI sound more authoritative than the evidence allows.

Important Implementation Details

The arithmetic did not change: Dirtydash still computes cached reads over reported prompt input plus reported cache tokens. The change is the contract. The UI and API now describe the value as cache_read_share, and the dashboard keeps cache writes visibly tied to source-reported fields.

Expected Impact for End-Users

Users should read the 95% figure as “most reported input tokens were served from cache,” not as proof that Dirtydash knows every cache creation event. The zero write count now communicates absence of reported write data rather than absence of caching behavior.

Validation

  • Ran npm run build in dashboard/.
  • Ran cargo test.
  • Opened the served dashboard in the in-app browser and confirmed the Cache page copy renders without overflow at narrow and desktop breakpoints.
  • Smoke-checked /api/summary: it returns both cache_read_share and hit_ratio with matching values.

Issues, Limitations, and Mitigations

This still cannot reconstruct cache writes from sources that do not log them. The mitigation is honest labeling and retaining the write field only for sources that actually report cache creation/write tokens.
