Summary

The main discrepancy came from DirtyDash treating Codex per-request rows as cumulative deltas. A second issue appeared after fixing that: Codex cached input is included inside input_tokens, so DirtyDash had to split cached input out before applying separate cache-read pricing.

Changes Made

  • Added Codex-specific usage extraction for payload.info.last_token_usage and payload.info.total_token_usage.
  • Used last_token_usage directly when present, matching CodexBar and Tokscale accounting.
  • Kept total_token_usage as the fallback cumulative source for degraded or older rows.
  • Split Codex cached input into cache_read_tokens and non-cached prompt_tokens before cost estimation.
  • Updated importer regression coverage with the nested Codex JSONL shape.

Context

The user observed DirtyDash around $459, CodexBar around $1241 for the past 30 days, and Tokscale around $1.6k all-time. A read-only comparison against local Codex logs, CodexBar, and Tokscale showed that DirtyDash was importing a similar number of Codex rows but storing much smaller token values.

Important Implementation Details

  • serde_json iterates object keys in sorted order, so the old recursive extractor often found last_token_usage before total_token_usage.
  • The old parser then applied saturating_delta to that per-request value, undercounting later rows.
  • CodexBar and Tokscale both treat cached input as part of reported input, then split it into non-cached input plus cache-read tokens for cost math.
  • DirtyDash still does not model CodexBar's Priority/Fast surcharge or long-context rates, so a remaining difference from CodexBar is expected.

Issues, Limitations, and Mitigations

  • Known difference: CodexBar includes Priority/Fast and long-context pricing paths that DirtyDash still lacks.
  • DirtyDash's existing user database must be re-imported or rebuilt to replace previously undercounted Codex rows.
  • The bundled pricing table is static. Tokscale can use live or cached pricing catalogs, so exact parity still depends on rate-source alignment.
