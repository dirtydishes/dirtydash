Add dirtydash deployment script and docs

Added a repository-owned deployment path for updating the existing dirtydash server installation from either this Mac or directly on di.

Issue: dirtydash-j6d Date: 2026-06-08 Scope: script and docs

Summary

This change adds scripts/deploy-dirtydash and docs/deployment.md. The script rebuilds the dashboard and Rust binary on the server, recreates the private Docker container, and smoke-tests the NPM upstream. The documentation explains local deploys, server-side deploys, manual SQLite sync, validation, configuration, and rollback.

Changes Made

  • Created an executable Bash deploy script with remote SSH mode and direct --local server mode.
  • Added flags for remote host, app directory, data directory, branch selection, and --skip-pull.
  • Documented the current di deployment layout and NPM upstream.
  • Documented the manual SQLite sync flow while keeping machine-local credentials out of the repository.

Context

dirtydash currently runs behind Nginx Proxy Manager as a private Docker container on the npm-shared network. Before this change, redeploying required copying commands from the previous turn record. This gives the repository a repeatable update path.

Important Implementation Details

  • The deploy script does not change NPM configuration or Basic Auth credentials.
  • The script does not sync the SQLite database by default; dirtydash-sync-db remains the manual DB sync path.
  • Remote mode sends the deployment commands over SSH, so it works before the script itself has been pulled onto the server.
  • Server mode uses --local for cases where the script is already present on di.

Deploying dirtydash updates should now be a single command instead of a remembered sequence. Remote readers can also understand the current server layout and validation steps from repository docs.

Issues, Limitations, and Mitigations

  • The script assumes the existing di deployment shape, including Docker, Rust, Node, and the npm-shared network. The docs list the overrides for different paths or names.
  • App-native authentication is still not implemented. NPM Basic Auth remains the current protection.
  • The script recreates the container instead of building a versioned image. This matches the current lightweight deployment and can be replaced by an image-based deploy later.

Follow-up Work

  • Add app-native auth for dirtydash before broadening access.
  • Consider a Dockerfile and tagged image flow once deploy frequency increases.
  • Optionally add an automated database sync job if manual sync becomes annoying.
