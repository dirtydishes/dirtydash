Why This Was Done

The goal was to get dirtydash online quickly while keeping the app private enough for usage and cost data. Running it as a Docker container on NPM’s shared network avoids exposing port 4599 publicly and gives NPM a stable upstream name.

Issues, Limitations, and Risks

  • The dashboard is protected by NPM Basic Auth, not app-native dirtydash auth.
  • Manual sync replaces the server SQLite file with a backup copy from this Mac. It does not merge multiple machines.
  • The dirtydash container mounts the compiled binary from the host. Rebuilding the app on the server updates the mounted binary path, but the container may need a restart to pick up a new binary.
  • A disabled user service file remains at /home/delta/.config/systemd/user/dirtydash.service. It is not enabled or running.

Rollback or Recovery

To stop and remove the deployed container:

ssh di 'docker rm -f dirtydash'

To remove the local sync command:

rm -f ~/bin/dirtydash-sync-db

To remove the NPM route, delete proxy host ID 15 and access list ID 2 in Nginx Proxy Manager. If doing it from the UI, remove the proxy host first, then the access list.

To redeploy after a repo update:

ssh di 'cd ~/apps/dirtydash/app && git pull --ff-only && cd dashboard && npm ci && npm run build && cd .. && ~/.cargo/bin/cargo build --release && docker restart dirtydash'
