Restore Ledger Panels Inside Fixed Viewport

Fixed the regression where the Ledger view could collapse into a session-only surface after the viewport lock. The old four-panel Ledger composition is back, while document scrolling stays disabled.

2026-06-18 03:22 dirtydash-16d dashboard regression fix

Summary

The Ledger page now explicitly places its four panes inside the fixed viewport: daily ledger and graph on top, selected-day sessions below them, and the inspector on the right. Medium and narrow breakpoints also receive explicit placements so panels are not auto-placed out of view.

Changes Made

  • Restored meaningful Ledger row proportions with minmax(300px, 1fr) and minmax(220px, 0.68fr).
  • Changed .sessions-pane from a global full-span rule to a Ledger-specific placement.
  • Placed .ledger-pane, .chart-pane, .sessions-pane, and .inspector-pane explicitly in the Ledger grid.
  • Adjusted the 1180px breakpoint so Ledger uses a two-by-two panel grid instead of hiding later panels.
  • Adjusted the 820px breakpoint so all four panels remain visible in one fixed-height column.
  • Rebuilt dashboard/dist so the embedded Rust-served dashboard uses the fixed CSS.

Context

The previous viewport fix correctly stopped page scrolling, but it changed the Ledger grid from an old scroll-tolerant layout into a clipped fixed-height layout. Auto-placement was too implicit for that constraint. The correction keeps the fixed app shell but makes Ledger panel placement deterministic.

Important Implementation Details

Ledger owns session placement .sessions-pane no longer globally spans all columns. Inside Ledger it spans columns one and two, leaving the inspector column intact.
Desktop keeps the old shape The graph is back beside the ledger table, sessions are below them, and inspector fills the right side.
Medium widths show all panels Below 1180px, Ledger becomes a two-by-two grid: ledger, chart, sessions, inspector.
Viewport lock remains The outer shell still measures to the viewport, and arrow-key navigation does not scroll the document.

Expected Impact for End-Users

The Ledger page once again looks like the Ledger page: graph, daily table, selected-day sessions, and inspector are all visible. Users still get the intended fixed-screen behavior, so arrow keys can navigate selection without browser page movement.

Validation

  • Ran npm run build in dashboard/. Vite completed successfully.
  • Ran cargo test. All 21 Rust tests passed.
  • Served the embedded dashboard with cargo run -p dirtydash -- serve --port 4609.
  • Used headless Chrome DevTools checks at 1280x720, 1100x720, and 390x844.
  • Confirmed all four Ledger panels reported visible bounding boxes at each viewport.
  • Confirmed documentElement.scrollHeight and body.scrollHeight matched viewport height at each checked size.
  • Confirmed window.scrollY remained 0 after pressing ArrowDown.

Issues, Limitations, and Mitigations

  • The narrow one-column layout necessarily gives each panel less height, but all four panels stay visible inside the fixed frame.
  • Dense table content still scrolls inside panes. That is intentional and avoids returning to document-level page scrolling.
  • The browser verification used Chrome DevTools directly because this repository does not have Playwright installed.
