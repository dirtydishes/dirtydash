Preserve Codex Fast Model Totals

Dirtydash now resolves Codex low-effort GPT 5.5 and GPT 5.4 sessions to their fast pricing slugs before storing usage events, so the dashboard can show fast-mode spend and tokens as separate model rows.

Summary

Fast-mode Codex usage was being imported as the base model, usually gpt-5.5, because the parser ignored session reasoning effort. The importer now treats low-effort GPT 5.5 and GPT 5.4 Codex turns as gpt-5.5-fast and gpt-5.4-fast.

Changes Made

  • Added reasoning-effort extraction for Codex session and event payloads.
  • Resolved gpt-5.5 and gpt-5.4 to fast slugs when effort is fast, low, minimal, off, or none.
  • Added a regression test that imports both GPT 5.5 low-effort and GPT 5.4 minimal-effort Codex token-count records.
  • Re-imported the local database, updating existing rows whose resolved model and pricing changed.

Context

The dashboard screenshot showed a large gpt-5.5 row but no gpt-5.5-fast row. The pricing snapshot already included fast records, and the model aggregation already preserved distinct model slugs. The missing piece was importer normalization: Codex stores the base model and the reasoning effort separately in turn context payloads.

Important Implementation Details

  • The normalization is intentionally narrow and only applies to gpt-5.5 and gpt-5.4.
  • Explicit model slugs such as gpt-5.5-fast remain unchanged.
  • Existing database rows can be repaired by re-running import because Dirtydash updates an existing usage event when the raw event hash matches but pricing/model fields differ.
  • The local re-import updated 22,314 existing events and inserted 156 new events.

fn model_with_reasoning_effort(model: &str, effort: Option<&str>) -> String {
    let model = model.trim();
    let Some(effort) = effort else {
        return model.to_string();
    };
    let effort = effort.trim().to_ascii_lowercase();
    if matches!(effort.as_str(), "fast" | "low" | "minimal" | "off" | "none")
        && matches!(model, "gpt-5.5" | "gpt-5.4")
    {
        format!("{model}-fast")
    } else {
        model.to_string()
    }
}

Expected Impact for End-Users

Users should now see fast-mode GPT 5.5 and GPT 5.4 usage separated from the base model totals after importing Codex sessions. Estimated costs also use the fast pricing rows when those rows match the session effort.

Validation

  • cargo fmt --check passed.
  • cargo test -p dirtydash passed: 10 unit tests and 2 CLI tests.
  • npm --prefix dashboard run build passed.
  • cargo run -p dirtydash -- import --json completed with 581 files seen, 156 inserted events, 22,314 updated existing events, 2,913 skipped events, and 0 parse errors.
  • cargo run -p dirtydash -- doctor --json reported 29,135 events, 18 pricing records, 3 detected sources, and no warnings.
  • SQLite verification showed gpt-5.5-fast at $923.1192 across 449,698,176 tokens and gpt-5.4-fast at $19.0825 across 30,050,773 tokens.

Issues, Limitations, and Mitigations

  • Some older gpt-5.4-fast events still carry the earlier 2026-06-06-bundled pricing version, which reflects previously imported rows rather than a dashboard display failure.
  • The effort-to-fast mapping is based on observed Codex traces and current bundled pricing assumptions. If Codex introduces a separate explicit fast slug or a new effort vocabulary, this helper may need another small update.
  • The change does not rewrite raw session files; it only changes imported model labels and cost estimates.

Follow-up Work

  • No follow-up Beads issue was created because the requested GPT 5.5 and GPT 5.4 fast-mode split is implemented and validated.
  • A future enhancement could expose reasoning effort as a separate dashboard dimension, but that is not required for the current model totals fix.
