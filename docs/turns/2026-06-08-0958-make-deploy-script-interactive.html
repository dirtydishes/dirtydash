Make deploy script interactive

Updated the dirtydash deployment helper so normal use prompts for remote versus local deployment instead of requiring the user to remember mode flags.

Issue: dirtydash-yojDate: 2026-06-08Scope: script UX and docs

Summary

scripts/deploy-dirtydash now asks whether to deploy remotely over SSH or locally from the server shell when no explicit mode is supplied. Remote remains the default when the user presses Enter.

Changes Made

  • Added an interactive choose_mode prompt.
  • Added --remote-mode for automation and non-interactive callers.
  • Kept --local as the explicit server-side mode.
  • Updated deployment docs with prompt behavior and examples.

Context

The first deploy script worked, but the most natural command still required knowing whether to pass --local. This change makes the default path conversational while preserving deterministic flags for CI, scripts, and non-interactive shells.

Important Implementation Details

  • Non-interactive stdin now fails clearly unless --remote-mode or --local is supplied.
  • The prompt writes to stderr so command output remains cleaner.
  • Invalid answers exit before any deploy work starts.

Expected Impact for End-Users

Running scripts/deploy-dirtydash is easier to understand in the moment. Users can choose remote or local deployment from the prompt, while automation still has explicit mode flags.

Issues, Limitations, and Mitigations

  • Interactive local mode was not used for a real deploy from this Mac because it is intended for the server shell. The explicit remote path and prompt parser were validated separately.
  • The script still assumes the existing di Docker/NPM deployment shape. The docs list environment overrides for path and naming changes.
