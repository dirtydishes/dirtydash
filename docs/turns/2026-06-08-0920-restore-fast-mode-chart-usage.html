Restore Fast Mode Chart Usage

Fast/priority usage is visible again in Dirtydash charts, with yellow bar segments showing the fast portion of token and cost usage where the data can prove it.

Summary

Updated the dashboard and pricing reconciliation so fast mode usage is restored and consistently visible across relevant usage views.

Changes Made

  • Added metric-aware chart bars so token charts highlight fast tokens and cost charts highlight fast spend.
  • Added the daily token usage chart back to Overview and added a model spend chart to Burn Report.
  • Expanded Pricing rows to show both token and cost bars, each with yellow fast/priority segments when present.
  • Fixed legacy *-fast model reclassification so those rows become priority usage instead of standard usage.
  • Added trace-based repair for imported Codex turns with priority evidence in nearby logs_2.sqlite.

Context

The API already exposed priority_tokens and priority_estimated_cost_usd, but recent pricing cleanup could normalize old fast model rows without preserving their priority mode. Existing charts also only rendered a token-oriented segment, so pricing/cost views did not clearly show the fast portion.

Important Implementation Details

  • UsageBar now accepts a tokens or cost metric and computes the yellow segment from the matching priority field.
  • Bundled pricing seeding now repairs legacy -fast rows as PricingMode::Priority.
  • For already-imported Codex rows with turn IDs, Dirtydash can re-read local trace evidence and restore priority mode without a reset.
  • Rows that were previously imported without turn IDs cannot be proven priority from the database alone; re-importing with the current parser is the recovery path for those.

Expected Impact for End-Users

Users can scan the same model, source, daily usage, burn, and pricing views and immediately see how much of each bar came from fast/priority mode. Pricing rows also expose the fast spend amount directly.

Validation

  • Ran npm run build in dashboard/.
  • Ran cargo test; all Rust and CLI tests passed.
  • Verified rendered overview and pricing bars in the in-app browser using an isolated temporary database with standard and priority usage rows.

Issues, Limitations, and Mitigations

  • The current local database had Codex rows without turn_id, so it could not prove fast mode for those already-normalized rows. The mitigation is to re-run import with the current parser, which can update rows with turn IDs when source logs still contain them.
  • Priority repair only touches non-manual and non-reported-cost rows, preserving user overrides and reported costs.

Follow-up Work

  • Add a small doctor warning when Codex rows lack turn IDs, because that blocks priority repair.
  • Consider a one-shot CLI repair command that explains how many rows can be updated before applying changes.
