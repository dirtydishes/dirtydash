Single-Screen Keyboard Selection

Locked the Dirtydash dashboard into a single viewport and changed arrow-key behavior so the keys move interface selection instead of scrolling the document.

2026-06-18 03:13 dirtydash-zv9 dashboard keyboard ux

Summary

The app shell now fills the viewport and hides document-level overflow. The dashboard content area remains inside that fixed frame, while panes handle their own overflow. Arrow keys are reserved for moving the row cursor, day context, or active tab, so the page does not jump while navigating sessions.

Changes Made

  • Added sorted session collections in dashboard/src/main.tsx so the selected session, highlighted row, and inspector all follow the same display order.
  • Mapped ArrowUp and ArrowDown to the row cursor, with j and k kept as matching shortcuts.
  • Mapped ArrowLeft and ArrowRight to day changes on Ledger and tab changes elsewhere.
  • Updated the shortcut overlay to document the new arrow-key behavior.
  • Changed dashboard/src/styles.css so html, body, #root, .tui-shell, and .dashboard use a fixed viewport-height frame with hidden document overflow.
  • Rebuilt dashboard/dist so the Rust-embedded dashboard bundle matches the source changes.

Context

Dirtydash is meant to feel like a compact local instrument for terminal-heavy users. In that context, arrow keys should operate the dashboard selection model. Browser page scrolling makes the interface feel less like a TUI and also means the user can lose the active session area while trying to inspect rows.

Important Implementation Details

Selection follows visible order The app sorts session arrays before using them as the keyboard cursor source, so the inspector does not drift away from the row that appears selected.
Document scroll is disabled The page-level containers use fixed height and hidden overflow. Individual panes retain overflow: auto for dense tables and long paths.
Arrow keys are consumed The keyboard handler calls preventDefault() for arrow navigation outside text inputs, which stops the browser from scrolling the page.
Mobile stays framed The narrow layout keeps the rail above a fixed-height dashboard area, preserving the no-document-scroll rule.

Expected Impact for End-Users

Users can keep their eyes on the dashboard while moving through sessions with the arrow keys. The app no longer behaves like a web page that scrolls away from the active row; it behaves more like the compact local dashboard the product is aiming for.

Validation

  • Ran npm run build in dashboard/. Vite completed successfully.
  • Ran cargo test. All 21 Rust tests passed, including CLI tests.
  • Started cargo run -p dirtydash -- serve --port 4609 and inspected the embedded app in headless Chrome through the DevTools protocol.
  • At 1280x720, Chrome measured documentElement.scrollHeight=720, body.scrollHeight=720, .tui-shell=720, and .dashboard=720.
  • At 1280x720, pressing ArrowDown moved the active session index from 0 to 1 while window.scrollY stayed 0.
  • At 390x844, Chrome measured document height equal to viewport height and reported no console, runtime, or network errors.

Issues, Limitations, and Mitigations

  • Dense panes can still scroll internally. That is intentional, because tables and long paths can exceed the available panel area while the app shell itself stays fixed.
  • On narrow screens, the app preserves the single-screen constraint, but some panes may feel compressed. The mitigation is pane-level overflow rather than document scrolling.
  • The browser smoke test used Chrome headless through the DevTools protocol because Playwright is not installed in this repository.
