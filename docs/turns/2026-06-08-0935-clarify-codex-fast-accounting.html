Clarify Codex Fast Accounting

Turn record for dirtydash-4wh, generated June 8, 2026 at 09:35.

Summary

Dirtydash now explicitly separates API-style token cost estimates from Codex subscription fast-mode limit drain. The dashboard no longer leaves users to infer that cache-read pricing explains fast-mode usage.

Changes Made

  • Added a Codex limit drain metric on Burn Report with a clear not measured value.
  • Added a Burn Report accounting note explaining that xhigh fast-mode sessions can drain a separate subscription ledger.
  • Added a Pricing note clarifying that rates are per 1M tokens and cache-read pricing is not Codex fast-mode subscription usage.
  • Added a small reusable AccountingNote component and matching table-note styling.
  • Rebuilt the checked-in dashboard bundle so the embedded server serves the updated UI.

Context

The reported screenshot showed a low dollar figure for fast usage even though recent Codex sessions were mostly xhigh and fast mode. The important distinction is that Dirtydash estimates cost from imported token categories, while subscription fast-mode drain can be counted by Codex using a different ledger that local logs do not currently expose as a reliable field.

Important Implementation Details

  • No importer math was changed. Cached input is still split from prompt tokens and priced with the cache-read rate when public-style pricing exists.
  • The UI now treats Codex subscription limit drain as unknown rather than trying to infer it from cached input tokens.
  • The copy names output and reasoning tokens because those can be significant in xhigh sessions and should not be hidden behind cache-read accounting.

Expected Impact for End-Users

Users should be less likely to mistake a low cache-read cost estimate for their real Codex subscription limit consumption. The dashboard now says plainly when a number is not available instead of implying precision that the logs cannot support.

Issues, Limitations, and Mitigations

This change does not calculate true Codex subscription fast-mode drain because the current imported session logs do not expose a trustworthy fast-mode counter. The mitigation is explicit labeling: the dashboard now marks the value as not measured and points users away from treating cache-read cost as fast-mode usage.
