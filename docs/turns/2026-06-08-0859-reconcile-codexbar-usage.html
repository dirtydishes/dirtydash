CodexBar Usage Reconciliation

Dirtydash now follows CodexBar more closely for Codex source coverage, priority pricing semantics, and token-summary display. The main divergence was not one thing: archived sessions were outside default scanning, priority pricing was treated as a multiplier, and the dashboard did not make cached input semantics visible.

Summary

Added archived Codex session discovery, replaced blanket fast/priority multiplication with CodexBar-style priority rate handling, preserved completed-response model evidence for priority pricing, and made dashboard token summaries show input including cache separately from generated tokens.

Changes Made

  • Codex source detection now includes sibling archived_sessions roots next to sessions.
  • Codex priority evidence now reads priority request rows and completed response model rows from logs_2.sqlite.
  • Pricing now uses explicit CodexBar priority rates for GPT-5.5, GPT-5.4, and GPT-5.4-mini, with the 272k input-token cap.
  • Grouped usage API rows now include reasoning_tokens, and dashboard rows show total, input including cache, generated output, and fast/priority tokens.
  • Rebuilt the dashboard bundle so the shipped static assets match the source changes.

Context

The user reported impossible fast-mode attribution and a token-total mismatch: CodexBar showed roughly billion-scale usage while the dashboard could appear much smaller. CodexBar documentation and source showed that its local scan includes ~/.codex/sessions, ~/.codex/archived_sessions, pi-agent sessions, trace database priority metadata, and model-aware pricing.

Important Implementation Details

  • Reasoning effort no longer implies fast mode. Priority classification comes from trace evidence only.
  • Priority pricing is not a generic multiplier. GPT-5.4 priority in particular would be wrong under multiplier-only math.
  • When trace evidence says a priority request completed as a different model, Dirtydash uses that completed model for pricing but keeps the original session model bucket visible.
  • CodexBar cache arrays are effectively [input including cached, cached subset, output]. Dirtydash stores uncached prompt and cached read separately, then displays their combined input explicitly.

Expected Impact for End-Users

Users should see Codex usage in the same order of magnitude as CodexBar after reimporting, and fast/priority attribution should reflect real service-tier evidence instead of reasoning-effort labels. The dashboard now explains why huge cached-input sessions can look different depending on whether a tool displays input including cache or cached reads as a separate subset.

Validation

  • Passed: npm --prefix dashboard run build.
  • Passed: temp full import: 649 files seen, 28,831 events inserted, 0 parse errors.
  • Observed: updated scan reports both ~/.codex/sessions and ~/.codex/archived_sessions as Codex sources.
  • Observed: browser check against temp DB rendered 2.8B total tokens, 2.8B input including cache, and 39.4M fast tokens.
  • Note: dirtydash doctor --json was run once against the real app database, which may apply schema migrations and pricing metadata seeds. The real database was not reimported during this turn.

Issues, Limitations, and Mitigations

  • Remaining parser nuance: CodexBar still has more sophisticated fork-baseline and divergent cumulative-total handling than Dirtydash. Mitigation: filed dirtydash-ejj.
  • Existing DB state: users need to run a fresh import for archived-session discovery and revised pricing modes to update stored rows.
  • Validation side effect: the real app database may already have the new migration/pricing metadata from the doctor command, but usage rows still reflect the last full import until the user refreshes them.
  • CodexBar cache freshness can still shift reconciliation numbers slightly, because its local cache may not represent the exact same scan instant as a fresh Dirtydash import.
