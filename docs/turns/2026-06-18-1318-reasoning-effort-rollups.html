Reasoning Effort Rollups

Implemented Codex reasoning-effort detection as explicit usage metadata, added token-weighted backend rollups, and exposed the breakdown in dense dashboard bars and session rows.

ScopeImporter, SQLite schema, API structs, aggregation, dashboard UI, tests, embedded dashboard assets.
Primary outcomeReasoning effort is now auditable as configured effort, separate from model reasoning token counts.
Validationcargo test, cargo build, npm run build, browser inspection, contrast check.

Summary

Dirtydash now stores normalized reasoning effort on usage events and per-turn records. Daily and session responses include reasoning buckets weighted by total_tokens, with missing metadata shown as unknown.

Changes Made

  • Added ReasoningEffort, normalization rules, raw evidence storage, and Codex turn_context extraction.
  • Added usage_events.reasoning_effort, usage_events.raw_reasoning_effort, reasoning_turns, and supporting indexes.
  • Added reasoning buckets to NamedUsagePoint and SessionSummary.
  • Added the chart bars: usage | reasoning control and a compact session-table Reasoning column.
  • Updated generated dashboard assets under dashboard/dist/ so the embedded server serves the new UI.

Context

The existing reasoning_tokens field means model-produced reasoning token count. This task needed a separate configured-effort dimension, so the implementation keeps token accounting unchanged and introduces explicit reasoning_effort buckets for rollups.

Important Implementation Details

  • minimal, none, off, false, and 0 normalize to none, displayed as “no reasoning.”
  • Unrecognized or missing values normalize to unknown, while raw strings remain available for audit.
  • Duplicate turn contexts prefer the latest non-unknown value in reasoning_turns.
  • Bucket shares are computed from summed total_tokens, and zero-token buckets are omitted.
  • Migration backfills null or blank effort values to unknown so partially migrated local databases remain readable.

Expected Impact for End-Users

Users can now see whether usage came from no, low, medium, high, xhigh, or unknown configured reasoning effort. Daily bars can switch from usage coloring to reasoning coloring, and session rows expose a concise reasoning summary without hiding the existing token and cost columns.

Validation

  • cargo test: 23 library tests, 2 CLI tests, and doc tests passed.
  • cargo build: completed successfully.
  • npm run build in dashboard/: completed successfully and regenerated embedded assets.
  • Manual browser inspection at http://127.0.0.1:5179: overview reasoning toggle, reasoning bars, sessions reasoning column, unknown rows, and narrow viewport horizontal table scroll verified.
  • Contrast check: reasoning text colors against --panel ranged from 6.21:1 to 10.16:1.

Issues, Limitations, and Mitigations

  • Existing imported rows without explicit metadata show unknown. This is intentional and makes parser coverage visible.
  • Prompt-level UI is not exposed yet. The new reasoning_turns table stores the per-turn evidence needed for that later.
  • Reasoning buckets are based on token totals, not prompt counts, matching the locked product decision.

Follow-up Work

  • Optionally add a future prompt or turn inspector that reads from reasoning_turns.
  • Consider a small legend for reasoning colors if users find the stacked bars too terse.
  • After reimporting Codex sessions with turn metadata, spot-check that historical rows move from unknown to known buckets where source logs support it.
