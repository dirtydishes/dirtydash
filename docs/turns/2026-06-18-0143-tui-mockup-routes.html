TUI-inspired mockup routes

Added four standalone Dirtydash design mockups at /mock1, /mock2, /mock3, and /mock4. They lean toward an enhanced TUI workflow with panes, command context, keyboard shortcuts, dense rows, and inline inspectors instead of conventional dashboard cards.

Issue: dirtydash-0g1 Date: 2026-06-18 Scope: React mock routes, CSS, bundled dashboard build

Summary

The dashboard now has four visual studies for a developer-focused Dirtydash interface: Run Ledger, Source Matrix, Session Inspector, and Local Ops. Each route works without backend API data, supports shared hotkeys, and keeps the current product’s local-first usage/cost/provenance domain visible.

Changes Made

  • Added route detection for /mock1 through /mock4 before the normal API-backed dashboard renders.
  • Built mock datasets for sessions, sources, and models so the mockups can run from the Vite dev server or bundled static app without API responses.
  • Added a shared mock shell with route shortcuts, command context, sort controls, view controls, active-row cursor state, and an inspector rail.
  • Added route-specific surfaces for ledger review, source/model matrix inspection, session search/provenance, and local ops/runbook workflows.
  • Regenerated the bundled dashboard assets under dashboard/dist/.

Context

Dirtydash is for technical developers who already trust terminal workflows, but the requested direction was not a literal terminal clone. The mockups use terminal-native concepts like command buffers, cursors, panes, and key hints while preserving web strengths: responsive layout, clickable controls, readable inspectors, sticky structure, and safe horizontal scrolling for dense data.

Important Implementation Details

  • s cycles sorting by cost, tokens, and cache; v cycles view modes; j/k move the active session; number keys jump between mock routes.
  • The normal app still fetches live API data on existing routes. Mock routes skip that loading path so they remain stable design artifacts.
  • The Rust server already falls back to index.html, so direct static requests for /mock1 and friends resolve to the React app.
  • Responsive fixes keep the page itself from horizontally overflowing while preserving internal scroll for dense table panes.

Expected Impact for End-Users

Users and reviewers can now compare four concrete directions for the next Dirtydash UI without disturbing the existing dashboard. The strongest ideas are the persistent inspector, visible hotkey model, dense pane layout, and provenance-first session handling.

Validation

  • Ran npm run build in dashboard/.
  • Opened /mock1, /mock2, /mock3, and /mock4 through local Chrome automation.
  • Verified desktop headings and text did not overflow on all four routes.
  • Verified hotkeys changed state: s switched sort, v switched view, and j moved the selected session.
  • Verified mobile page width on /mock3 stayed contained at 390px while the session pane retained internal horizontal scrolling.

Issues, Limitations, and Mitigations

  • The mockups use static sample data by design. This keeps the routes deterministic, but the next implementation pass would need to bind these layouts to real dashboard data.
  • The visual direction intentionally uses dense panes and horizontal data scrolling. Mobile containment was validated, but a production version should decide which tables become condensed summaries on very small screens.
  • The root package-lock.json file was already untracked and was not included in this work.

Follow-up Work

  • Choose the preferred direction or combine the best pieces into one production route.
  • Wire hotkeys to a real command/search model, including / focus and accessible shortcut documentation.
  • Move mock-only data into fixtures if these routes stay around as a design lab.
