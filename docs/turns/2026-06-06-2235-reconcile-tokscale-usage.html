DirtyDash now reconciles Codex, pi-agent, and OpenCode usage with tokscale.

The mismatch was not a UI formatting issue. DirtyDash was missing source coverage and treating Codex token snapshots with the wrong event semantics, which suppressed cache-read volume and cost.

Summary

Reworked DirtyDash import accounting so Codex token-count rows, archived Codex sessions, modern OpenCode SQLite messages, reported tool costs, and model aliases are all represented in the local SQLite dashboard. After re-import, the app database lines up with tokscale within the small drift caused by new session rows being written during validation.

Changes Made

  • Changed Codex token-count parsing to use payload.info.last_token_usage as the per-event increment and total_token_usage only for baseline and stale-snapshot checks.
  • Split Codex cached input out of prompt input before storing and pricing, matching tokscale’s non-cached input plus cache-read accounting.
  • Added default source coverage for ~/.codex/archived_sessions and ~/.local/share/opencode/opencode.db.
  • Added OpenCode SQLite parsing, numeric timestamp handling, reported-cost extraction, and per-file stale row cleanup on re-import.
  • Extended bundled pricing for gpt-5.3-codex-spark, gpt-5.4-fast, opencode/big-pickle, and openai-codex provider fallback.
  • Added regression tests for Codex last-usage semantics and OpenCode SQLite ingestion.

Context

The tokscale TUI showed roughly 2.7B tokens and about $1.6K all time, while DirtyDash and CodexBar were much lower. Direct inspection showed tokscale scans active Codex sessions, archived Codex sessions, pi-agent sessions, and OpenCode’s SQLite database. DirtyDash only had partial coverage and had stale imported rows from the earlier parser.

The largest numerical error came from Codex cache-read accounting. Codex logs contain mutable cumulative snapshots and a per-request last_token_usage block. DirtyDash’s old recursive extractor found the per-request values and then subtracted the previous per-request row, which collapsed many cache-read rows toward zero.

Important Implementation Details

  • Codex rows now store prompt tokens as non-cached input, cache-read tokens separately, output tokens, and reasoning tokens. This avoids double-counting cached input while still making total local usage visible.
  • Re-imports update existing raw-event rows and delete stale rows for each successfully parsed file. That cleanup matters because the old parser produced rows that the corrected parser intentionally skips.
  • OpenCode SQLite rows are treated as first-class source files. The legacy JSON message directory is still scanned when present.
  • Rows with source-reported nonzero costs keep pricing_version = reported-cost. Otherwise DirtyDash uses bundled pricing and provider aliases.

Expected Impact for End-Users

DirtyDash should no longer understate Codex spend by hundreds of dollars or hide most of the cached-token volume. Users should see OpenCode and archived Codex activity reflected automatically after import, and re-running import should repair stale parser output in place.

Validation

  • Automated: cargo test passed with 6 unit tests, 2 CLI integration tests, and doc tests.
  • Fresh import: temp database import loaded 28,008 events, 2.674B tokens, and about $1.622K before later live-log growth.
  • Real app DB: re-imported /Users/kell/Library/Application Support/dev.dirtydash.dirtydash/dirtydash.sqlite3 with 603 files and no parse errors.
  • Tokscale comparison: DirtyDash all-time was $1,624.21 and tokscale was $1,624.40 in the final adjacent snapshots. DirtyDash local-day was $64.54 and tokscale local-day was $64.73.

Issues, Limitations, and Mitigations

  • Live Codex logs changed while validation ran, so adjacent DirtyDash and tokscale snapshots can differ by a few current-turn rows. The remaining observed gap was less than one dollar and moved with active logging.
  • DirtyDash still skips zero-token messages. Tokscale includes some zero-token OpenCode/pi message counts, so message counts can differ while token and cost totals align.
  • CodexBar may still report a different 30-day value if it applies service-tier, fast/priority, or long-context pricing semantics that DirtyDash does not yet model.
