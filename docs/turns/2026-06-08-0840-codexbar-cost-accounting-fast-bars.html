CodexBar-Aligned Cost Accounting With Fast Usage Bars

Dirtydash now prices Codex usage with CodexBar-style local accounting, records the pricing mode on each event, and shows locally proven priority traffic as yellow fast segments in the shared usage bars.

Summary

The headline remains Estimated spend, but Codex estimates now come from deterministic local pricing modes instead of reasoning-effort model rewrites. Dirtydash stores turn_id and pricing_mode per usage event, uses local logs_2.sqlite rows as the only proof of priority service tier, applies long-context pricing above the CodexBar threshold, and renders yellow priority/fast segments wherever aggregate token bars are shared.

Changes Made

  • Added turn_id and pricing_mode to usage events and aggregate API rows, including standard_tokens, priority_tokens, and priority_estimated_cost_usd.
  • Removed reasoning-effort based *-fast model rewriting for Codex imports; reasoning effort remains source metadata, not a pricing tier.
  • Read local Codex priority evidence from CODEX_HOME/logs_2.sqlite, nearby session trace databases, or ~/.codex/logs_2.sqlite, matching priority rows to JSONL token events by turn_id.
  • Seeded CodexBar-style standard, long-context, and priority pricing for Codex models while preserving manual overrides and local free flags.
  • Reclassified old non-overridden bundled *-fast and *-long usage rows back to base model rows with explicit pricing modes.
  • Updated the shared bar component and Pricing page so priority usage appears as a yellow token-based segment with a text label such as 110K fast.

Context

The previous implementation used Codex reasoning effort to synthesize fast model slugs. That made model totals diverge from the way CodexBar accounts locally: priority/fast cost is a service-tier property proven in Codex trace metadata, while model name and reasoning effort remain separate. The new path does not require login, OAuth, OpenAI APIs, or network calls.

Important Implementation Details

  • pricing_mode can be reported, manual, free, standard, long-context, priority, or unpriced.
  • Priority is applied only when a matching local trace row has service_tier: "priority" for the same turn_id.
  • Long-context pricing starts above 272_000 total input tokens, using Dirtydash’s invariant that prompt_tokens is uncached input and cache_read_tokens is cached input.
  • Cache reads remain separately priced and clamped; cache writes stay at the seeded CodexBar rate, currently zero for these bundled Codex rows.
  • The shared UsageBar handles Overview, Models, Projects, sink/source totals, cache-style breakdown rows when mode data exists, and Pricing usage cells.

Expected Impact for End-Users

  • Codex spend should be closer to CodexBar-style local accounting without asking users to authenticate or call remote APIs.
  • Users can see how much of a model, source, or project total came from locally proven priority/fast mode by reading the yellow segment and the adjacent fast label.
  • Rows with no priority evidence keep the same violet-only visual treatment, so standard usage remains quiet and familiar.
  • The Pricing page now connects model rates to imported usage volume while keeping provenance as source/model/snapshot metadata.

Issues, Limitations, and Mitigations

  • If logs_2.sqlite is missing or lacks a matching priority row, Dirtydash deliberately falls back to standard or long-context pricing and shows no yellow fast segment.
  • Historical rows created under the old fast/long model-slug scheme are repaired when bundled pricing seeds run, but custom manual overrides still keep precedence.
  • The broad CodexBar 30-day reconciliation is still tracked separately in dirtydash-q9g; this turn implemented the deterministic accounting machinery and UI surfaces.
